## The NULL Method – How It Works

You add `UNION SELECT` with a certain number of NULL values. If the count is wrong, the database errors. If the count is right, it succeeds.

## Simple Step-by-Step

**Try 1 NULL:**
```sql
' UNION SELECT NULL --
```
Error → wrong count

**Try 2 NULLs:**
```sql
' UNION SELECT NULL, NULL --
```
Error → wrong count

**Try 3 NULLs:**
```sql
' UNION SELECT NULL, NULL, NULL --
```
No error → correct count

**Conclusion:** Original query returns **3 columns**

## Why Use NULL?

| Problem | Solution |
|---------|----------|
| You don't know column types | NULL works with any type (text, number, date) |
| Different data types cause errors | NULL avoids type mismatch errors |

## What Success Looks Like

When the NULL count matches:

| Application behavior | What you see |
|---------------------|---------------|
| Shows query results | An extra blank row appears |
| Processes data | Different error (like NullPointerException) |
| Ignores extra rows | Response looks the same as before |

## The Limitation (You Mentioned)

Sometimes the response looks identical whether you succeed or fail. In this case, **the NULL method doesn't work** – use the ORDER BY method instead.

## Comparison of Both Methods

| ORDER BY | UNION SELECT NULL |
|----------|-------------------|
| `' ORDER BY 1 --` | `' UNION SELECT NULL --` |
| Works by causing errors | Works by matching column count |
| Always reliable | Can fail if app ignores extra rows |
| Increment until error | Increment until no error |

## One Sentence Summary

Add NULLs one at a time until the error disappears – that number tells you how many columns the original query returns.
