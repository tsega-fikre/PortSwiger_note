## 1. The Big Idea So Far

Until now, we talked about SQL injection happening in the **URL** or in a **cookie**.

Example URL:
```
https://website.com/search?product=laptop
```

The `product=laptop` part is called the **query string**. That's where we've been injecting our attacks.

---

## 2. The New Idea: Different Delivery Methods

Websites don't just take input from URLs.

Sometimes they take input from:

- **Forms** you fill out
- **JSON** (a way computers package data, like a digital envelope)
- **XML** (another way computers package data, like a different style of envelope)

The important part: **All of these can be vulnerable to SQL injection.**

---

## 3. The Problem: Security Guards (WAFs)

Websites often have a **security guard** called a **WAF** (Web Application Firewall).

The WAF looks at everything coming into the website and says:

> *"Hmm... if I see the word **SELECT** or **UNION** or **OR 1=1**, I'm going to **block this request**."*

It's like a bouncer at a club checking IDs and looking for trouble.

---

## 4. The Trick: Speaking in Code (Encoding)

If the bouncer is looking for the word **SELECT**, what if you **disguise** the word **SELECT**?

Instead of sending:
```
SELECT
```

You send it in a **secret code** that the bouncer doesn't recognize, but the **database does understand**.

---

## 5. The Example Explained (XML Escape Sequence)

Here's the example from the text:

```xml
<stockCheck>
    <productId>123</productId>
    <storeId>999 &#x53;ELECT * FROM information_schema.tables</storeId>
</stockCheck>
```

Let's break this down **word by word** like we're in kindergarten.

---

### Part 1: What is this format?

```xml
<stockCheck>
    <productId>123</productId>
    <storeId>999 something here</storeId>
</stockCheck>
```

This is **XML**. Think of it as a **labeled box**.

| XML Tag | Meaning |
|---------|---------|
| `<stockCheck>` | "This is a stock check request box" |
| `<productId>123</productId>` | "The product ID is 123" |
| `<storeId>999 something</storeId>` | "The store ID is 999 something" |

---

### Part 2: The Disguised Word

Look closely at:
```
&#x53;ELECT
```

That `&#x53;` is the **secret code**.

---

### Part 3: How the Code Works

In computers, every letter has a **number**.

| Letter | Number (in a system called ASCII) |
|--------|-----------------------------------|
| A | 65 |
| B | 66 |
| S | **83** |

Now, **83** in **hexadecimal** (a fancy way of counting used by computers) is **53**.

So:
```
&#x53;  =  S
```

The `&#x` means: *"The next two characters are a hexadecimal number. Turn that number back into a letter."*

---

### Part 4: The Full Translation

| What the Attacker Sends | What It Really Means |
|-------------------------|----------------------|
| `&#x53;ELECT` | `SELECT` |

So:
```
999 &#x53;ELECT * FROM information_schema.tables
```

Becomes:
```
999 SELECT * FROM information_schema.tables
```

---

## 6. Why This Works (The Bouncer Analogy)

Imagine the **WAF bouncer** has a list of banned words:
- `SELECT`
- `UNION`
- `DROP`

When the request arrives, the bouncer reads:
```
999 &#x53;ELECT * FROM ...
```

The bouncer thinks:

> *"Hmm, I see `&#x53;ELECT`. That's not on my banned list. It doesn't say `SELECT`. Okay, you can go in."*

The bouncer **lets it through**.

---

## 7. What Happens Inside the Club (The Server)

Once past the bouncer, the request goes to the **server**.

The server says:

> *"Oh, this is XML. Let me **decode** it first."*

It sees `&#x53;` and thinks:

> *"`&#x53;` is code for the letter `S`. Let me turn that back into `S`."*

Now it becomes:
```
999 SELECT * FROM information_schema.tables
```

The server then **passes this to the database**.

The database sees `SELECT` and says:

> *"Okay, I'll run this query!"*

**Attack successful.**

## 9. Other Ways to Disguise Words

Attackers have **many costumes** for banned words:

| Method | SELECT disguised as | Works in |
|--------|---------------------|----------|
| XML Hex | `&#x53;ELECT` | XML |
| URL Encoding | `%53ELECT` | URLs |
| Unicode | `\u0053ELECT` | JSON |
| Case Switching | `SeLeCt` | Anywhere (if WAF is dumb) |
| Comments Inside | `SEL/**/ECT` | SQL |

---

## 10. Simple Summary in One Sentence

> **The attacker writes the banned word in a secret code that the security guard doesn't understand, but the database does understand after the code is translated.**

---

## 11. Analogy for a Child

You want to tell your friend a **secret word** (`SELECT`), but the teacher is listening and will punish you if she hears `SELECT`.

So you **spell it out** using a code:

> *"S is 83 in secret number language. 83 in special code is 53. So I'll say `&#x53;` instead of S."*

You whisper: `&#x53;ELECT`

The teacher hears gibberish and ignores you.

Your friend knows the code, translates `&#x53;` back to `S`, and understands you said `SELECT`.
